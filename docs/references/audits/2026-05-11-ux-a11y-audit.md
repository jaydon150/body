# UX / Accessibility audit — Phase 1 vertical slice

**Date:** 2026-05-11
**Auditor:** UX/Accessibility agent (Tier 2)
**Dispatch:** P1.18
**Scope:** WCAG 2.2 AA conformance of the running Phase 1 vertical slice at `app/web/` (P1.01 → P1.17 complete).
**Method:** Static analysis of `app/web/src/`, computed contrast ratios via WCAG sRGB→linear formula, visual inspection of P1.17 rendering baselines (`tests/rendering-snapshots/baseline-{desktop,ipad-landscape,ipad-portrait}.png`), keyboard-flow walkthrough on the source.

---

## One-line summary

**Conditional pass: WCAG 2.2 AA is now met after the fixes applied in this dispatch, with two caveats deferred to a design decision** — (a) long-press visual feedback during the 500 ms touch-dive press is missing and warrants a follow-up (UA-009); (b) `prefers-reduced-motion` outside the dive lerp has been audited and is fine, but Phase 2 should re-audit when new JS-driven motion lands (e.g. peel transitions, focus highlights).

Pre-fix, the slice would have failed AA on at least four counts (UA-001 muted-text contrast, UA-005 reduced-motion dive, UA-007 scene-host focus visibility, UA-006 search dialog name). All four are fixed in this dispatch.

- L = 0.2126·R' + 0.7152·G' + 0.0722·B' where each channel C' = C/255 ≤ 0.03928 ? C/12.92 : ((C/255 + 0.055)/1.055)^2.4
- Contrast ratio = (max(L1, L2) + 0.05) / (min(L1, L2) + 0.05)

## Reduced-motion analysis

**Pre-fix:** the dive camera lerp ran unconditionally over 600 ms. The global `@media (prefers-reduced-motion: reduce)` rule in `index.css:984-989` zeroes `animation-duration` + `transition-duration` for all elements, which DOES cover the 200 ms iPad sidebar slide-over and any CSS transitions in peel-controls / nomenclature-toggle / search trigger / about trigger / sidebar rows. It does NOT cover the JS `useFrame` lerp in `CameraRig.tsx` — `useFrame` calls into Three.js per requestAnimationFrame and is invisible to CSS.

**Post-fix:** [`CameraRig.tsx`](../../../app/web/src/scene/CameraRig.tsx) now:
- Imports a `usePrefersReducedMotion()` hook (defined inline in the same file) that subscribes to `window.matchMedia('(prefers-reduced-motion: reduce)')` with `addEventListener('change', …)` so toggles during a session propagate.
- In the dive `useEffect`, if `reducedMotion` is true: clear `animRef.current`, snap `camera.position` + `controls.target` to the target pose, re-enable controls, return early.
- The `useFrame` lerp consumer only runs while `animRef.current !== null`, so a snap'd transition completes immediately and the lerp body becomes a no-op.

**Long-press timer (500 ms):** is a *wait*, not motion — out of scope for `prefers-reduced-motion`. Confirmed.

## Long-press feedback (UA-009) — recommendation

**Problem:** A user touches a bone on iPad. Behind the scenes, `LONG_PRESS_MS = 500` is counting down. The user has NO visual signal:
- That a long press is being detected.
- How much time remains before the dive triggers.
- Whether moving their finger > 8 px will cancel.
- What the difference between "tap to select" and "long-press to dive" is.

A user may release at 400 ms thinking nothing is happening, or hold long enough that an accidental orbit gesture turns into a dive.

**Recommended fix (deferred; needs design decision before implementation):**

1. **Radial-progress ring** rendered as a CSS-only `<div>` positioned absolutely above the canvas at the pointer's `clientX/Y`. On `pointerdown` (touch only), the ring mounts with `transform: scale(0)` and a CSS transition `transform 500ms linear`. The ring scales to full size as the timer counts down. If `pointerup` / cancel-by-move fires before 500 ms, the ring unmounts with a fade. Color: `--accent-cyan`, low opacity, ~24-32 px diameter.

   ```css
   .long-press-ring {
     position: absolute;
     width: 32px; height: 32px;
     border-radius: 50%;
     border: 2px solid var(--accent-cyan);
     opacity: 0.4;
     pointer-events: none;
     transform: translate(-50%, -50%) scale(0);
     transition: transform 500ms linear, opacity 120ms ease-out;
   }
   .long-press-ring--active { transform: translate(-50%, -50%) scale(1); }
   ```

2. **Reduced-motion fallback:** when `prefers-reduced-motion: reduce`, replace the radial scale with a simple opacity step (0 → 0.5 → 1.0) — still conveys "something is happening" without rotation/scale motion.

3. **Wire-up:** `SkeletalScene.tsx`'s `onPointerDown` already tracks `(startX, startY, entryId, pointerId)` in `longPressStateRef`. On a touch press, render an additional ring element via React portal at those coordinates; remove on `cancelLongPress`.

**Effort:** ~40 LoC TSX + ~30 lines CSS + a state addition (a ref or local state to drive ring visibility).

**Why deferred:** Although the fix is localised, it involves a design judgement on the *form* of the feedback (ring vs flash vs bar) that I would prefer not to make unilaterally. UA-009 is **the one item the orchestrator must decide on before claiming the slice is shippable to actual users on iPad.**
