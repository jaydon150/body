# tests/review-queue

Anatomist review queue artefacts. One folder per batch, named
`<date>-batch-<n>/`. Each batch folder is generated by
`pipelines/07-anatomist-review/generate-packet.mjs` and consumed
(after the human anatomist has marked it up) by
`pipelines/07-anatomist-review/promote.mjs`.

## Workflow

```
                 queue                       human review           promote
    [pending records]  --> [batch folder]  -------------->  [edited manifest]  --> [reviewed records]
   (data/canonical/      (tests/review-       (out-of-band)    (manifest.json    (promote.mjs writes
   ontology/content/)     queue/<batch>/)                       fields filled)    confidence='reviewed')
```

The cadence is intentionally batched: anatomist time is scarce; we
amortise the context-switch cost by reviewing tens of records in a
sitting rather than one at a time.

## Per-batch contents

A batch folder, e.g. `2026-05-11-batch-1/`, contains:

- `review-packet.md` — Markdown read by the anatomist. One section
  per content record: preferred label, structure_id, FMA alias, TA
  Latin name, summary, long-form prose, citations, four-checkbox
  reviewer-action stub, free-text notes blockquote. Closes with a
  signature block.
- `manifest.json` — machine-readable queue state. Updated by hand (or
  by a future Content/QA dispatch helper) as the anatomist works
  through the packet. Consumed by `promote.mjs`.

## Manifest shape (schema-by-example)

```json
{
  "version": "2026-05-11",
  "batch_id": "2026-05-11-batch-1",
  "generated_at": "2026-05-11T00:00:00Z",
  "anatomist": {
    "name": "Jane Doe, PhD",
    "credentials": "Professor of Anatomy, University X",
    "contact": "jdoe@university.example"
  },
  "status": "queued",
  "items": [
    {
      "structure_id": "UBERON:0000981",
      "filename": "uberon_0000981.json",
      "preferred_label": "femur",
      "status": "queued",
      "queued_at": "2026-05-11T00:00:00Z",
      "reviewed_at": null,
      "decision": null,
      "reviewer_notes": null
    }
  ]
}
```

### Top-level fields

- `version` — schema version of the manifest itself. Currently
  `2026-05-11`. Bump when the manifest shape changes.
- `batch_id` — matches the folder name.
- `generated_at` — ISO-8601 datetime the packet+manifest were generated.
- `anatomist` — reviewer identity. `name` is **required** before the
  promotion script will run (the content-record schema requires
  `reviewed_by` for any `reviewed` record).
- `status` — overall batch state. One of:
  - `queued` — generated, anatomist has not started.
  - `in_review` — anatomist has begun marking items.
  - `complete` — anatomist has signed off; manifest ready for
    `promote.mjs`.
- `items` — array, one per content record in the batch.

### Per-item fields

- `structure_id` — UBERON/FMA/BODY ID, mirrors the content record.
- `filename` — file under `data/canonical/ontology/content/` to
  promote.
- `preferred_label` — human label for the manifest reader's
  convenience (sourced from `nodes.json` at generate time).
- `status` — review decision. One of:
  - `queued` — not yet reviewed.
  - `in_review` — anatomist is currently reading this one.
  - `approved` — record stands as authored; `promote.mjs` will
    promote to `reviewed`.
  - `approved_with_edits` — record is fine in substance but the
    prose needs Content-agent revision before final sign-off;
    `promote.mjs` does **not** auto-promote.
  - `rejected` — record is wrong / out of scope; stays `pending`
    pending a Content-agent rewrite or removal.
  - `needs_research` — anatomist could not sign off without primary
    sources; escalates to Research/Docs.
- `queued_at` — ISO-8601 datetime set at generate time.
- `reviewed_at` — ISO-8601 datetime; set by the anatomist (or helper)
  when status moves off `queued`.
- `decision` — short string summarising the decision. **Required**
  when `status === "approved"`; `promote.mjs` refuses to promote a
  record with an empty decision.
- `reviewer_notes` — optional free text; if non-empty for an approved
  record it is copied into `review_notes` on the content record.

## Open item — JSON Schema for the manifest

A formal JSON Schema for `anatomist-review-manifest.json` would live
under `app/shared/schema/` and be wired into the main project
validator. That is the **Architect's** authority and was deliberately
not done in this dispatch. Proposed as an open item for a future
Architect dispatch — the schema-by-example above is the de-facto
contract until then.

## Why Markdown for v1 of the packet

A `.md` file:
- diffs cleanly under version control,
- renders nicely on GitHub for review threading,
- prints to PDF via any browser if the anatomist prefers paper,
- requires zero additional pipeline dependencies (the script is
  Node built-ins only).

A real PDF generator (Puppeteer, LaTeX) is a heavier dependency and
is reserved for Phase 2+ if the anatomist requests a formatted PDF.
